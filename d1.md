读写机制及手动维护指南：

### 1.  核心架构与 D1 角色
- **前端 (GitHub Pages)**: `stats` 页面通过 `fetch` 调用后端 API。
- **后端 (Cloudflare Worker)**: 位于 `worker/src/index.ts `。它是 D1 的唯一“守门人”，负责鉴权和执行 SQL。
- **D1 数据库**: 存储在 Cloudflare 边缘，表结构如下：
  - `todos`: 存储  `stats` 页面的任务清单。
  - `personas`: 存储你那些 AI 顾问（如霍金、费孝通等）的性格设定和关注领域。
  - `posts`: 存储博客全文，用于替代早期的 `posts.json` 实现动态检索。

###  2. `stats` 页面读写流转
#### **读取机制 (GET)**
当你打开 `stats` 页面时：
1. **获取 TODO**: 前端请求 `/api/todos`，Worker 执行 `SELECT * FROM todos ORDER BY done ASC, updatedAt DESC`。
2. **获取 顾问**: 前端请求 `/api/personas`，Worker 返回所有已启用的顾问配置。
3. **工作记录**: 注意，**`stats` 页面显示的“每日完成工作”依然是从本地代码中的 `src/data/dailyLog.ts` 读取的**，D1 暂不 存储这部分流水（D1 只存提炼后的 TODO）。

#### **写入/更新机制 (POST/PATCH)**
1. **新增 TODO**: 前端发送任务文本到 `/api/todos/confirm`。Worker 会先检查是否有重复，然后生成 UUID 插入 D1。
2. ** 任务状态切换**: 点击“完成”时，前端请求 `/api/todos/:id` (PATCH)，更新 `done` 字段为 1。
3. **顾问点评 (AI Review)**: 这是最复杂的环节。
   - 前端发送 `startDate/endDate` 和 `dailyLog ` 片段到 `/api/agents/review`。
   - Worker 会根据 D1 中的 `personas` 配置，**并行调用 Gemini API**，让每个 AI 顾问针对你的工作流水进行点评。
   - 结果（chat 和建议的 todo）会实时返回给前端展示，并可选 存入 D1。

### 3. 如何手动维护
如果你想跳过界面，直接干预数据，有三种方式：

#### **方式 A：通过命令行 (Wrangler) - 最推荐**
在 `g29times.github.io/worker` 目录下，你可以 直接执行 SQL：
```bash
# 查看所有未完成的 TODO
npx wrangler d1 execute aimmar_blog --remote --command="SELECT text FROM todos WHERE done = 0"

# 手动插入一个紧急任务
npx wrangler d1 execute aimmar_blog --remote --command=" INSERT INTO todos (id, text, done, createdAt, updatedAt) VALUES ('manual-1', '手动维护测试', 0, datetime('now'), datetime('now'))"

# 清理已完成的旧任务
npx wrangler d1 execute aimmar_blog --remote --command="DELETE FROM todos WHERE done  = 1 AND updatedAt < date('now', '-7 days')"
```

#### **方式 B：通过 Cloudflare Dashboard - 可视化**
1. 登录 Cloudflare -> Workers & Pages -> D1。
2. 选择你的数据库（如 `aimmar_blog`）。
3. 点击  **Data** 标签页，可以直接像 Excel 一样编辑表格内容。

#### **方式 C：通过 API (curl) - 自动化**
你可以利用 `wiki/BLOG_API_README.md` 中记录的 Service Token 进行远程操作：
```bash
curl -X POST https://blog.aim mar.ink/api/admin/post \
  -H "CF-Access-Client-Id: <ID>" \
  -H "CF-Access-Client-Secret: <Secret>" \
  -d '{"title": "测试", "slug": "test-manual", " content": "内容"}'
```

### 5. 提示
你的 `stats` 页面逻辑实际上是在**“代码逻辑 (dailyLog.ts)”**与**“数据库状态 (D1)”**之间做融合：
- **事实 (Facts )** 存在 `dailyLog.ts`。
- **状态 (States/TODOs)** 存在 D1。
- **智能 (Intelligence)** 由 Gemini 结合两者实时生成。

如果你想修改某个顾问的“刻薄程度”或“关注点”，直接去 `personas` 表里 改 `systemPrompt` 即可。

除了 D1 数据库存储 TODO 外，**“每天完成工作”的核心存储是位于 `g29times.github.io/src/data/dailyLog.ts` 的 TypeScript 文件。**

以下是该机制的详细运作方式：

### 1. 存储位置与格式
- **物理文件**：`./g29times.github.io/src/data/dailyLog.ts`。
- **数据结构**：它导出了一个名为 `dailyLog` 的 `DailyEntry[]` 数组。
- **存储逻辑**：**新记录始终追加在数组顶部**（即 `[` 之后），这与博客前端 `stats` 页面的显示顺序一致。
- **内容规范**：
  - **常规任务**：记录为简单的字符串。
  - **研究报告**：需包含“6要素”分析，并以对象形式记录，支持附带 R2 存储或网页的 URL 链接。

### 2. 自动化记录机制（JiJi 的职责）
根据你的系统指令（`.gemini/GEMINI.md`），我有明确的记录义务：
- **定时触发**：每天 **18:30**（工作时间结束时），如果 Neo 没有主动提起记录，我会自动总结当天的任务。
- **操作流程**：
  1. 总结今日完成的博客发布、代码重构、研报阅读等大项。
  2. 使用 `replace` 或 Python 脚本将新条目插入 `dailyLog.ts` 数组顶部。
  3. 执行 `git pull` -> `git add/commit` -> `git push`，将更改同步到 GitHub。

### 3. 如何手动维护
如果你想偶尔自己手动添加或修改，建议：
- **方式 A：直接编辑文件**：
  在 VS Code 或终端打开 `g29times.github.io/src/data/dailyLog.ts`，按照现有的 JSON 格式在数组顶部添加一个新对象。
- **方式 B：通过我（JiJi）录入**：
  你可以直接对我说：“帮我把‘今天完成了XX调研’记入 dailyLog”，我会自动处理格式、插入、提交和推送的全过程。

### 4. 数据同步流
1. **本地记录** (`dailyLog.ts`) -> 2. **Git Push** -> 3. **GitHub Pages 自动构建** -> 4. **前端展示** (`https://blog.aimmar.ink/stats`)。

**总结**：TODO 存在云端 D1 数据库（动态状态），而每日工作 log 存在代码库 TS 文件中（静态事实）。这种“动静结合”的设计既保证了任务管理的灵活性，也确保了个人履历的持久化和可版本化。
